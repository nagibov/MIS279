Lecture 3.1

Why We Need Bridge Tables

‚ùå WRONG WAY
Orders Table (Bad Design)
Order_ID: 1, Product_IDs: "101,102,103"
Order_ID: 2, Product_IDs: "101,105"
Problems:
  • Can't query individual products easily
  • Can't store quantities per product
  • Violates database normalization rules
‚úÖ CORRECT WAY
Order_Details Table (Good Design)
Order_ID: 1, Product_ID: 101, Quantity: 2
Order_ID: 1, Product_ID: 102, Quantity: 1
Order_ID: 1, Product_ID: 103, Quantity: 3
Order_ID: 2, Product_ID: 101, Quantity: 1
Benefits:
  • Easy to query and filter
  • Stores additional data (quantity, price)
  • Follows normalization principles
Lecture 3.1

Schema Best Practices

1 Avoid Redundancy (Normalization)

Don't repeat the same data in multiple places. Use foreign keys to reference related data.

‚ùå Redundant Design
Order_ID: 1, Customer_Name: "John", Customer_Email: "john@email.com"
Order_ID: 2, Customer_Name: "John", Customer_Email: "john@email.com"
‚úÖ Normalized Design
Orders: Order_ID: 1, Customer_ID: 101
Customers: Customer_ID: 101, Name: "John", Email: "john@email.com"
2 Use Indexes for Speed

Add indexes on frequently searched columns (email, product_name, order_date).

3 Foreign Keys for Integrity

Ensure data consistency by using foreign key constraints between related tables.

Discussion Time

Let's Think About It (5 minutes)

1 What happens if we don't normalize a product catalog?

Think about: duplicate data, storage waste, update anomalies...

2 Why separate "Orders" and "OrderDetails"?

Consider: order-level info vs. product-level info...

3 Should we store passwords in plain text?
NEVER! Always hash passwords using bcrypt or similar secure methods.

ERD Workshop

Step 4: Normalization Exercise

4 Let's fix a common problem: Customer Addresses
‚ùå BEFORE (Denormalized)
Orders Table
Order_ID: 1, Customer_ID: 101
Ship_Street: "123 Main St"
Ship_City: "Boston"
Ship_State: "MA"
Ship_ZIP: "02101"
Problem: If a customer moves, we have to update their address in every order record!
‚úÖ AFTER (Normalized)
Orders Table
Order_ID: 1, Customer_ID: 101
Shipping_Address_ID: 501
Addresses Table
Address_ID: 501, Customer_ID: 101
Street: "123 Main St", City: "Boston"
State: "MA", ZIP: "02101"
Benefit: Update address once, affects all related orders automatically!
